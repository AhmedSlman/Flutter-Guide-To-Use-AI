01 — قراءة التصميم عبر Figma MCP

أول حاجة في أي مشروع — الـ Agent يقرأ الـ Figma ويعمل Design Map بالكامل

Figma MCP · مرة واحدة في بداية المشروع
⚠️
البرومبت ده بيطلب Map بس — لا كود. وافق على الـ Map وبعدين انتقل للخطوة التانية. ده بيمنع إن الـ agent يبني على فهم غلط.
📋prompt-01-read-design.txt
# ═══════════════════════════════════════════════
# PROMPT 01 — Read & Map Figma Design
# ═══════════════════════════════════════════════

You are a Senior Flutter Engineer working on this project.
Follow all project rules strictly. Respond in Egyptian Arabic.

## Task: Read Figma & Build Design Map

Using Figma MCP, connect to:
Figma URL: {{FIGMA_FILE_URL}}

### Step 1 — Design System Page ("_Design System")
Extract and map:
- All Color Variables  → names to use in AppColors.*
- All Typography Variables → names to use in predefined text styles
- All Spacing Variables → values for spacing constants
- All Radius Variables → names for border radius constants

### Step 2 — Components Page
List every Figma Component with:
- Exact layer name (snake_case)
- All Variants (states: default, loading, disabled, error, empty…)
- Component Properties (text labels, boolean toggles, etc.)
- Where it appears across screens (for reuse decisions)

### Step 3 — Screens Page
For every screen frame:
- Exact layer name → will become the View class name
- Identifiable Sections (logical groups of widgets)
- List of Component instances used
- Dev Mode annotations if present
- Navigation target (from prototype links)

### Output Required

DESIGN TOKENS:
  Colors:     [Figma variable name → AppColors.xxx]
  Typography: [Figma variable name → text style name]
  Spacing:    [variable → value in dp]
  Radius:     [variable → value in dp]

COMPONENTS:
  [layer_name]: variants=[...], props=[...], used_in=[...]

SCREENS MAP:
  [screen_layer_name]:
    suggested_view:     XxxView
    suggested_sections: [XxxSection, YyySection, ...]
    components_used:    [...]
    navigates_to:       [...]
    annotations:        [...]

QUESTIONS / UNCLEAR:
  [anything ambiguous in the design that needs clarification]

⛔ Do NOT write any Dart code in this step.
⛔ Do NOT create any files.
✅ Output only the structured map above.

03 — Feature Scaffold

بيجهز الـ feature folder كامل بنفس البنية الموجودة في المشروع

Feature · مكرر لكل Feature جديدة
💡
البرومبت ده بيعمل الـ boilerplate بس — cubit, model, repo, di, router, وstub views/sections/widgets. لا UI تفصيلي، لا API calls حقيقية. ده بييجي في الخطوات الجاية.
lib/features/{{FEATURE_FOLDER}}/ ├── cubit/ │ ├── feature_cubit.dart # Business logic only, connects repo ↔ sections │ └── feature_states.dart # Sealed/abstract states ├── data/ │ ├── model/ │ │ └── feature_model.dart # fromJson / toJson / copyWith │ ├── repo/ │ │ └── feature_repo.dart # Calls data_source, owns merging logic │ └── data_source/ │ └── feature_remote_ds.dart # Direct API calls via ApiConsumer ├── di/ │ └── feature_di.dart # GetIt registrations ├── router/ │ └── feature_router.dart # GoRouter routes, registered in core router └── presentation/ ├── views/ # Full screens — combine sections only │ └── feature_view.dart ├── sections/ # Logic bridge — connects cubit, passes data to widgets │ └── feature_section.dart └── widgets/ # Pure UI only — NO logic, NO cubit access └── feature_widget.dart
🧩prompt-03-feature-scaffold.txt
# ═══════════════════════════════════════════════
# PROMPT 03 — Feature Scaffold
# ═══════════════════════════════════════════════

You are a Senior Flutter Engineer on this project.
Follow all project rules. Respond in Egyptian Arabic.
MUST follow View → Section → Widget architecture strictly.

## Task: Scaffold New Feature

Feature Name (PascalCase): {{FEATURE_NAME}}
Feature Folder (snake_case): {{FEATURE_FOLDER}}
Views to create:    {{VIEW_LIST}}
Main entity fields: {{ENTITY_FIELDS}}
Repo methods:       {{REPO_METHODS}}

### Before You Start
1. Read lib/features/ to understand existing feature patterns
2. Read lib/core/locator/ to understand how DI is registered
3. Read lib/core/Router/ to understand how routes are added
4. Identify any existing shared widgets to reuse

### Files to Generate

cubit/{{FEATURE_FOLDER}}_states.dart
- Abstract or sealed base: {{FEATURE_NAME}}States
- States: Initial, Loading, Success(data), Error(message)
- Use copyWith pattern on Success state for granular updates
- Name: {{FEATURE_NAME}}Cubit, {{FEATURE_NAME}}States

cubit/{{FEATURE_FOLDER}}_cubit.dart
- Extends Cubit<{{FEATURE_NAME}}States>
- Constructor injects {{FEATURE_NAME}}Repo
- Business logic ONLY — no UI references
- Methods: one per repo method (e.g., fetch{{FEATURE_NAME}}s())
- Use ApiResult fold pattern:
  result.fold(
    onSuccess: (data) => emit({{FEATURE_NAME}}Success(data)),
    onFailure: (f) => emit({{FEATURE_NAME}}Error(f.userMessage(context))),
  );

data/model/{{FEATURE_FOLDER}}_model.dart
- Fields: {{ENTITY_FIELDS}}
- fromJson(Map json)
- toJson() → Map
- copyWith()

data/data_source/{{FEATURE_FOLDER}}_remote_ds.dart
- Constructor injects ApiConsumer
- Methods match repo interface
- Use ApiConsumer.get/post/put/delete
- Return raw ApiResult<{{FEATURE_NAME}}Model> (or List)
- All error handling via ErrorHandler.handleException()
- MOCK data for now (return static data), mark with // TODO: real API

data/repo/{{FEATURE_FOLDER}}_repo.dart
- Abstract class + implementation
- Implementation injects {{FEATURE_NAME}}RemoteDataSource
- Calls data source, maps result to domain model
- Owns merging logic if multiple sources needed

di/{{FEATURE_FOLDER}}_di.dart
- Register with GetIt:
  locator.registerLazySingleton<{{FEATURE_NAME}}RemoteDataSource>(...)
  locator.registerLazySingleton<{{FEATURE_NAME}}Repo>(...)
  locator.registerFactory<{{FEATURE_NAME}}Cubit>(...)
- Import and call this in core/locator/

router/{{FEATURE_FOLDER}}_router.dart
- GoRoute entries for each view in {{VIEW_LIST}}
- Route names as string constants in RouterNames or local constants
- Register routes in lib/core/Router/app_router.dart

presentation/views/ — one file per view in {{VIEW_LIST}}
- Wrap with BlocProvider(create: (_) => locator<{{FEATURE_NAME}}Cubit>()..fetch())
- Body: Column/Stack of Section widgets ONLY
- ZERO styling, ZERO BlocBuilder, ZERO direct logic
- Call fetch on initState or inside the BlocProvider create

presentation/sections/ — suggested sections from Figma Map
- Each section: one logical group of widgets
- Access cubit via context.read<{{FEATURE_NAME}}Cubit>()
- Use BlocBuilder/BlocConsumer with buildWhen to isolate rebuilds
- Use BlocSelector for single-field listening
- Pass ONLY needed data (not full state/model) to child widgets
- NO design code here — just logic bridging

presentation/widgets/ — one file per distinct UI element
- Pure StatelessWidget
- Receives all data via constructor parameters
- No cubit access, no context.read, no BuildContext logic
- const constructor where possible
- Max ~150 lines per file

## Strict Rules
⛔ No UI logic in Views
⛔ No styling/design code in Sections
⛔ No cubit access in Widgets
⛔ No BlocBuilder at the top of a View without buildWhen
⛔ No passing whole State or Model to a Widget that needs only 1-2 fields
⛔ No private helper methods like _buildXxx()
⛔ No ListView without .builder for dynamic lists
✅ Every file must compile with zero errors
✅ Run dart format . after generating

{{FEATURE_NAME}}PascalCase · مثال: ProductCatalog
{{FEATURE_FOLDER}}snake_case · مثال: product_catalog
{{VIEW_LIST}}مثال: ProductsView, ProductDetailView
{{ENTITY_FIELDS}}مثال: id: String, name: String, price: double, imageUrl: String?
{{REPO_METHODS}}مثال: getProducts(), getProductById(String id), searchProducts(String q)

View Class:    {{VIEW_CLASS_NAME}}

View File Rules:
- StatelessWidget or ConsumerWidget
- Wraps with BlocProvider if needed (use locator)
- Body is ONLY a Column/Stack of Section widgets
- ZERO styling, ZERO BlocBuilder, ZERO business logic
- AppBar built via existing AppCustomAppBar if available

Section File Rules:
- Access cubit: context.read<{{FEATURE_NAME}}Cubit>()
- Use BlocBuilder with buildWhen OR BlocSelector for single fields
- Handle ALL states: loading → shimmer, error → error widget, 
  empty → empty widget, success → pass data to widgets
- Pass ONLY required fields to each child Widget (not full model)
- NO design code, NO styling, NO decoration in sections
- Trigger events in initState via PostFrameCallback or section init

Widget File Rules:
- Pure StatelessWidget
- const constructor
- All data via constructor parameters
- NO BuildContext logic, NO cubit, NO navigation
- Use AppColors.*, AppSpacing.*, predefined text styles ONLY
- Max 150 lines — split if larger
- NO private helper methods like _buildXxx()

Performance Rules (from project rules):
- const everything possible
- BlocSelector for single-field listening
- buildWhen on every BlocBuilder
- RepaintBoundary around animated/complex widgets
- cached_network_image for all remote images with cacheWidth/cacheHeight
- ListView.builder (never ListView with children for dynamic lists)
- BoxDecoration with borderRadius instead of ClipRRect where possible
- Color.withOpacity() instead of Opacity widget

Localization:
- All user-facing strings via context.l10n.xxx
- NO hardcoded Arabic or English strings in widgets

### Phase 4 — Extraction
After main files, identify widgets that appear in multiple places.
Move them to:
- lib/core/common/widgets/ if reusable across features
- lib/shared/ if shared between a few features
- lib/features/{{FEATURE_FOLDER}}/presentation/widgets/ if feature-specific

2. View file — clean, no logic
3. Section files — logic only, no styling
4. Widget files — pure UI, no logic
5. Summary of what's mocked vs connected to real state

05 — API Collection Setup

بيقرأ ملف الـ Collection ويولّد كل الـ API Layer باستخدام ApiConsumer و ApiResult

Backend · مرة واحدة أو عند إضافة collection
💡
حط الـ Collection file في docs/api/ في المشروع. البرومبت ده بيولّد ApiEndpoints, Request Models, Response Models, وDataSources — كلها باستخدام الـ ApiConsumer وApiResult<T> الموجودين في المشروع.

# ═══════════════════════════════════════════════
# PROMPT 05 — API Collection Setup
# ═══════════════════════════════════════════════

You are a Senior Flutter Engineer on this project.
Follow all project rules. Respond in Egyptian Arabic.
Use ApiConsumer and ApiResult — never use Dio directly in data sources.

## Context
Collection File:  {{COLLECTION_FILE_PATH}}
Collection Type:  {{COLLECTION_TYPE}}
Base URL Env Key: {{BASE_URL_ENV_KEY}}
Auth Type:        {{AUTH_TYPE}}

### Step 1 — Analyze Before Coding
Read the collection file and output:
- List of all endpoints grouped by feature folder
- For each endpoint: method, path, request body fields, response schema
- Shared/reusable response wrappers (e.g., pagination wrapper, data wrapper)
- Auth requirements per endpoint group

⏸ Wait for my approval on the analysis before generating code.

### Step 2 — Update API Endpoints

lib/core/network/api_endpoints.dart (or equivalent)
Add:
  static const String endpointName = '/path/to/endpoint';
  // Group by feature. Add comment with HTTP method: // GET
  // NEVER hardcode base URL here

### Step 3 — Generate Models per Feature

Request Models
lib/features/{{feature}}/data/model/{{name}}_request_model.dart
- final fields, required in constructor
- toJson() → Map
- Use @JsonSerializable if project uses it, otherwise manual
- Optional fields are nullable

Response Models
lib/features/{{feature}}/data/model/{{name}}_model.dart
- fromJson(Map json)
- toJson() → Map  
- copyWith()
- Handle null safety carefully

Shared Response Wrapper (if API wraps all responses)
If all responses have a common wrapper like {data, message, status}:
Add to lib/core/network/ or reuse existing wrapper model.

### Step 4 — Generate Data Sources

lib/features/{{feature}}/data/data_source/{{feature}}_remote_ds.dart
For each endpoint:
  Future> methodName({{params}}) async {
    try {
      final response = await _apiConsumer.get(  // or post/put/delete
        ApiEndpoints.endpointName,
        queryParameters: {...},  // if GET
        data: requestModel.toJson(),  // if POST/PUT
        isFormData: {{IS_FORM_DATA}},  // true only for file uploads
      );
      return ApiResult.success({{Model}}.fromJson(response));
    } catch (e) {
      return ApiResult.failure(ErrorHandler.handleException(e));
    }
  }

### Step 5 — Update DI Registration
For each new model/datasource, ensure it's registered in the feature's di file.
No new GetIt calls in feature files directly.

### Step 6 — Environment Variables
Add to .env.example (ONLY .env.example — never real values in code):
{{BASE_URL_ENV_KEY}}=https://your-api.com/api/v1
Add any other env keys found in the collection.

## Rules
✅ Always use ApiConsumer interface — never import Dio directly in data sources
✅ Always use ApiResult.fold() pattern — never try/catch in Cubit
✅ Always use ApiEndpoints constants — never hardcode strings
✅ Use ErrorHandler.handleException() for all catches
✅ Use ErrorHandler.handleDioException() for Dio-specific errors
⛔ No auth tokens hardcoded anywhere
⛔ No direct Dio usage in data sources
⛔ No API calls in Repository — only in DataSource

You are a Senior Flutter Engineer on this project.
Follow all project rules. Respond in Egyptian Arabic.
Use ApiResult.fold() exclusively — no try/catch in Cubit.

## Target Feature
Feature Name:   {{FEATURE_NAME}}
Feature Folder: lib/features/{{FEATURE_FOLDER}}/
Endpoints:      {{ENDPOINT_LIST}}
Auth Required:  {{AUTH_REQUIRED}}
Pagination:     {{HAS_PAGINATION}}
Local Cache:    {{HAS_CACHE}}

### Step 1 — Audit (show results, no code yet)
Read these files and report:
- data_source: which methods are mocked → need real API
- repo: any direct API calls? (violation!) → fix first
- cubit: is fold pattern used? or try/catch? 
- sections: any API calls? (violation!) → fix first

⏸ Show audit results and wait for approval.

### Step 2 — Fix Violations First
If audit found any violations (API in repo, logic in widget, etc.)
fix them before wiring real API.

### Step 3 — Update Data Source

{{FEATURE_FOLDER}}_remote_ds.dart
Replace each // TODO: real API mock with:
  Future> methodName(params) async {
    try {
      final response = await _apiConsumer.get(
        ApiEndpoints.xxx,
        queryParameters: params.toMap(),
      );
      return ApiResult.success(T.fromJson(response['data']));
    } catch (e) {
      return ApiResult.failure(ErrorHandler.handleException(e));
    }
  }

### Step 4 — Update Repository
If repo was doing direct API calls (violation), fix:
- Remove any direct API code from repo
- Repo only calls _dataSource.method()
- Repo maps response model to domain if needed
- Repo handles data merging if {{HAS_CACHE}} = true:
  try remote first → on success cache it → on failure return cached

### Step 5 — Update Cubit (fold pattern)
Replace every method with the fold pattern:

  Future fetch{{FEATURE_NAME}}s() async {
    emit({{FEATURE_NAME}}Loading());
    final result = await _repo.get{{FEATURE_NAME}}s();
    result.fold(
      onSuccess: (data) => emit({{FEATURE_NAME}}Success(data)),
      onFailure: (failure) => emit({{FEATURE_NAME}}Error(failure.userMessage(context))),
    );
  }

For mutations (create/update/delete):
  emit({{FEATURE_NAME}}ActionLoading());
  result.fold(
    onSuccess: (_) {
      emit({{FEATURE_NAME}}ActionSuccess());
      fetch{{FEATURE_NAME}}s(); // refresh list
    },
    onFailure: (f) => emit({{FEATURE_NAME}}ActionError(f.userMessage(context))),
  );

### Step 6 — Pagination (if {{HAS_PAGINATION}} = true)
Add to Success state:
  - currentPage: int
  - hasMore: bool
  - items: List

Add to Cubit:
  - fetchNextPage() → append to existing list, increment page
  - refresh() → reset page to 1, replace list
  - Detect end: if response.length < pageSize → hasMore = false

In Section:
  - Listen to scroll via ScrollController
  - Call cubit.fetchNextPage() when near bottom
  - Use BlocSelector to watch hasMore separately from list

### Step 7 — Auth Token Check (if {{AUTH_REQUIRED}} = true)
Verify in lib/core/network/interceptors/:
- Token is attached from SecureStorage in the Dio interceptor
- On 401 → AuthFailure is thrown → triggers session logout
- Do NOT attach token manually in data source

### Step 8 — Verify UI States in Sections
Confirm each section handles:
- Loading → shimmer widget (BlocBuilder with buildWhen)
- Error   → existing error widget + retry button that calls cubit method
- Empty   → existing empty state widget  
- Success → renders real data from state
- No hardcoded strings → all via context.l10n

### Deliverables
1. Audit report [APPROVAL NEEDED]
2. Updated remote data source
3. Updated repository (if violations existed)
4. Updated cubit with fold pattern
5. Summary: endpoints wired, states covered, violations fixed

# ═══════════════════════════════════════════════
# PROMPT 07 — Modify Existing Feature
# ═══════════════════════════════════════════════

You are a Senior Flutter Engineer on this project.
Follow all project rules. Respond in Egyptian Arabic.
CRITICAL: Read existing code fully before touching anything.

## Change Request
Feature:         {{FEATURE_NAME}}
Feature Folder:  lib/features/{{FEATURE_FOLDER}}/
Change Type:     {{CHANGE_TYPE}}
Description:     {{CHANGE_DESCRIPTION}}
Figma Ref:       {{FIGMA_LAYER_OR_COMPONENT}}  (اتركه فاضي لو مش UI change)

# CHANGE_TYPE options:
# - ADD_SCREEN         → إضافة شاشة جديدة للـ feature
# - ADD_FIELD          → إضافة field في model أو form
# - MODIFY_UI          → تعديل تصميم شاشة موجودة
# - ADD_API_CALL       → إضافة endpoint جديد للـ feature
# - MODIFY_LOGIC       → تغيير business logic في الـ cubit
# - ADD_STATE          → إضافة state جديد للـ cubit
# - REFACTOR           → تحسين كود بدون تغيير سلوك

### Phase 1 — Read & Understand (لا تكتب كود)
Before ANY changes, read these files completely:

1. Feature Structure Scan
   - List all files inside lib/features/{{FEATURE_FOLDER}}/
   - Read cubit states — list all existing states and their fields
   - Read the cubit — list all methods and what they emit
   - Read the repo interface — list all methods and return types
   - Read each view — list sections it uses
   - Read each section — list widgets it renders and states it handles

2. Impact Analysis
   Based on {{CHANGE_DESCRIPTION}}, identify:
   - Files that WILL change
   - Files that MIGHT be affected (indirect impact)
   - Files that must NOT change
   - Shared widgets or core files that could be impacted

3. Change Plan
   Output a numbered plan:

   WILL CHANGE:
     [ ] file_path → what changes and why
     [ ] file_path → what changes and why

   MIGHT BE AFFECTED:
     [ ] file_path → why it might need updating

   WILL NOT TOUCH:
     [ ] file_path → reason

   NEW FILES NEEDED:
     [ ] file_path → purpose

⏸ Stop here. Wait for my approval on the plan before writing any code.

### Phase 2 — Implement (بعد الموافقة)

For ADD_SCREEN:
   1. Add new state(s) to existing states file (don't remove old ones)
   2. Add method(s) to cubit (don't touch existing methods)
   3. Add repo method + data source method (interface first, then impl)
   4. Register new route in feature_router.dart
   5. Create view → sections → widgets following V-S-W pattern
   6. Update DI if new dependencies needed

For ADD_FIELD:
   1. Add field to model (nullable if not required, add to fromJson/toJson/copyWith)
   2. If state holds the model → state's copyWith handles it automatically
   3. Update any widget that displays this field (pass as parameter)
   4. Update section to pass the new field to the widget
   5. If it's a form field → update validation in core/validation/

For MODIFY_UI:
   1. Use Figma MCP to read {{FIGMA_LAYER_OR_COMPONENT}}
   2. Identify which widget file(s) need changes
   3. Apply changes ONLY in widget files
   4. If layout changes → update section's structural arrangement
   5. Never add styling to sections

For ADD_API_CALL:
   1. Add endpoint to ApiEndpoints (if not already there)
   2. Add method to remote data source
   3. Add method to repo interface + implementation
   4. Add use-case-style method in cubit
   5. Add new states: XxxLoading, XxxSuccess, XxxError
   6. Wire section to handle the new states

For MODIFY_LOGIC:
   1. Read the current cubit method fully
   2. Understand all states it emits and who listens to them
   3. Make the minimal change needed
   4. Ensure all sections that use buildWhen still work correctly
   5. If state shape changes → update all BlocSelector calls

For REFACTOR:
   1. No behavior change — only structure/readability improvement
   2. Extract widgets if files > 150 lines
   3. Add buildWhen where missing
   4. Replace BlocBuilder with BlocSelector for single-field reads
   5. Add const where missing
   6. Confirm all tests/screens still work the same

### Phase 3 — Verify
After changes, confirm:
- All states are still handled in every section (no unhandled state)
- No new violations introduced (see forbidden list)
- Existing functionality not broken
- Run dart format .

### Deliverables
1. Impact plan [APPROVAL NEEDED]
2. Changed files only (don't resend unchanged files)
3. Summary: what changed, what was left untouched, any side effects

08 — Bug Fix / Debug

تحليل المشكلة وإصلاحها بدون ما تكسر حاجة تانية

Maintenance · عند ظهور أي bug أو سلوك غير متوقع
ℹ️
البرومبت ده فيه 3 مسارات — اختار الـ BUG_TYPE المناسب وهيتبع الـ checklist الخاص بيه. ده بيمنع الـ agent من الـ "سيبني هجرب حاجات" approach.
🐛prompt-08-bug-fix.txt
# ═══════════════════════════════════════════════
# PROMPT 08 — Bug Fix / Debug
# ═══════════════════════════════════════════════

You are a Senior Flutter Engineer on this project.
Follow all project rules. Respond in Egyptian Arabic.
CRITICAL: Diagnose first, fix second. Never guess.

## Bug Report
Feature:          {{FEATURE_NAME}}
Bug Type:         {{BUG_TYPE}}
Description:      {{BUG_DESCRIPTION}}
Steps to Repro:   {{REPRO_STEPS}}
Expected:         {{EXPECTED_BEHAVIOR}}
Actual:           {{ACTUAL_BEHAVIOR}}
Error Message:    {{ERROR_MESSAGE}}
Affected File(s): {{AFFECTED_FILES}}

# BUG_TYPE options:
# - UI_WRONG        → شاشة بتتعرض غلط / layout بينكسر
# - STATE_ISSUE     → state مش بيتحدث صح أو rebuild غلط
# - API_FAILURE     → API call بيفشل أو بيرجع بيانات غلط
# - NAVIGATION_BUG  → navigation مش شغال أو route غلط
# - CRASH           → الـ app بيتعطل مع error message
# - PERF_ISSUE      → الـ UI بطيء أو في jank أو memory leak
# - DATA_WRONG      → بيانات بتتعرض غلط أو مش بتتحدث

### Phase 1 — Diagnose (لا تكتب كود)

Step 1: Read the Error Trail
If {{ERROR_MESSAGE}} is provided:
- Parse the stack trace top-down
- Identify the exact file + line where the error originated
- Identify if it's a null safety error, type mismatch, or state error

Step 2: Read Relevant Files
Based on {{BUG_TYPE}}, read:

UI_WRONG:
  - The widget file where the issue appears
  - Its parent section (is it receiving correct data?)
  - Check: hardcoded values? wrong AppSpacing/AppColors key?
  - Check Figma MCP if needed to compare with design

STATE_ISSUE:
  - The cubit file — trace which method emits what state
  - All sections with BlocBuilder/BlocConsumer for this cubit
  - Check: is buildWhen filtering out the needed state?
  - Check: is BlocSelector watching the right field?
  - Check: is the state being emitted with correct data (copyWith)?

API_FAILURE:
  - The data source method that calls the API
  - The ErrorHandler.handleException path taken
  - The fold() in cubit — is onFailure emitting correctly?
  - Check: correct ApiEndpoints constant used?
  - Check: request model toJson() missing a field?
  - Check: response fromJson() crashing on null?

NAVIGATION_BUG:
  - Feature router file
  - Core app_router.dart — is route registered?
  - RouterNames constants — any typo?
  - The GoRouter.of(context).push/go call — correct path used?

CRASH:
  - Full stack trace analysis
  - Find the root cause file and line
  - Check for null access on nullable, type cast failure, or
    widget lifecycle issue (using context after dispose)

PERF_ISSUE:
  - Check for BlocBuilder without buildWhen at high level
  - Check for setState in large widget trees
  - Check for ListView with children (should be .builder)
  - Check for missing const constructors
  - Check for Opacity widget (use Color.withOpacity instead)
  - Check for ClipRRect inside list items
  - Check for image loading without cacheWidth/cacheHeight
  - Check for logic/calculations inside build()

DATA_WRONG:
  - Model's fromJson — check null handling and key names
  - Check if API response structure matches the model
  - Check if repo is mapping model correctly to entity
  - Check if cubit state copyWith is updating the right field

Step 3: Root Cause Report
Output:

ROOT CAUSE:
  File:    [file path]
  Line:    [approximate line]
  Reason:  [exact explanation of why the bug occurs]

CONTRIBUTING FACTORS:
  [any secondary issues found while reading the code]

FIX PLAN:
  [ ] Change X in file Y → reason
  [ ] Change A in file B → reason
  
WILL NOT TOUCH:
  [everything outside the fix scope]

⏸ Stop here. Wait for my confirmation before applying the fix.

### Phase 2 — Fix

Fix Rules:
- Change ONLY the files identified in the root cause
- Make the MINIMUM change needed to fix the bug
- Do NOT refactor unrelated code while fixing
- Do NOT "improve" other things you notice (create a separate task)
- Add a comment above the fix: // Fix: [brief explanation]

For STATE_ISSUE specifically:
Common fixes to check:
- Add/fix buildWhen: (prev, curr) => prev.xxx != curr.xxx
- Switch BlocBuilder → BlocSelector for single field
- Fix copyWith call that wasn't updating the right field
- Ensure emit() is not called with same state reference
- Check if BlocProvider is placed high enough in the tree

For API_FAILURE specifically:
Common fixes to check:
- Fix fromJson null safety: json['key'] as String? ?? ''
- Fix toJson missing field
- Fix wrong ApiEndpoints constant
- Fix wrong response['data'] vs response['result'] key
- Fix isFormData missing for file upload endpoint

For CRASH specifically:
Common fixes to check:
- Add null check: value ?? defaultValue
- Fix mounted check before setState/emit after async
- Fix type: (value as Map) 
- Fix late variable initialized too early/late

For PERF_ISSUE specifically:
Apply in order of impact:
1. Add buildWhen to all BlocBuilders in the affected screen
2. Convert ListView to ListView.builder
3. Add const to all static widgets
4. Add RepaintBoundary around heavy animated widget
5. Add cacheWidth/cacheHeight to all network images

### Phase 3 — Verify
After fix, confirm:
- The specific repro steps from {{REPRO_STEPS}} no longer trigger the bug
- No new violations introduced
- Surrounding code still works (no regressions)
- Run dart analyze — zero errors

### Deliverables
1. Root cause report [CONFIRMATION NEEDED]
2. Changed files only (minimum change)
3. Explanation of the fix in plain words
4. Any related issues found (to handle separately)
